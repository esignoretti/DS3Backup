<!-- API contracts the dashboard consumes -->

All API calls go to `http://127.0.0.1:{port}/api/v1/...` where port defaults to 8099.

GET /api/v1/status returns:
```typescript
interface StatusResponse {
    running: boolean;
    schedulerRunning: boolean;
    scheduledJobs: number;
    apiPort: number;
    uptime: string;
}
```

GET /api/v1/jobs returns:
```typescript
interface JobListResponse {
    jobs: BackupJobWithStatus[];
}
interface BackupJobWithStatus {
    id: string;
    name: string;
    sourcePath: string;
    retentionDays: number;
    objectLockMode: string;
    enabled: boolean;
    createdAt: string;  // ISO 8601
    lastRun: string | null;  // ISO 8601 or null
    nextRun: string;  // ISO 8601
    lastError: string;
    scheduleEnabled: boolean;
    cronExpr: string;
}
```

GET /api/v1/jobs/{id}/history returns:
```typescript
interface HistoryResponse {
    jobId: string;
    runs: BackupRun[];
}
interface BackupRun {
    jobId: string;
    runTime: string;  // ISO 8601
    status: string;   // "completed" | "failed" | "running"
    filesAdded: number;
    filesChanged: number;
    filesSkipped: number;
    filesFailed: number;
    bytesUploaded: number;
    batchesUploaded: number;
    duration: number;  // nanoseconds (JSON serialization of time.Duration)
    error: string;
    startTime: string;  // ISO 8601
    endTime: string;    // ISO 8601
}
```

POST /api/v1/backup/run/{id} — triggers a backup, returns 202:
```typescript
interface BackupTriggerResponse {
    jobId: string;
    triggered: boolean;
    message: string;
}
```

POST /api/v1/start — starts scheduler
POST /api/v1/stop — stops scheduler

      - "Run Backup" button — on click, calls POST /api/v1/backup/run/{id}
      - Collapsible "History" section showing the last N backup runs in a table

    ### History Table (per job)
    - Columns: Date/Time, Status (badge), Files Added, Files Changed, Size, Duration, Error
    - Colored status badges (green=completed, red=failed, yellow=running)

    ### Footer
    - Auto-refresh indicator (last updated timestamp)
    - "Refresh Now" button

    ## CSS Design

    - Dark theme: background #1a1a2e, cards #16213e, accent #0f3460, text #e0e0e0
    - Status colors: completed=#4caf50, failed=#f44336, running=#ff9800
    - Use system font stack: -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif
    - Cards with subtle box shadows and rounded corners (8px)
    - Responsive: flex-wrap on job cards, stack on narrow screens
    - Buttons with hover states
    - History tables: alternating row colors, scrollable on small screens
    - Transitions on collapsible history sections

    ## Important Implementation Details

    1. **Error handling**: If any API fetch fails, show a banner "Connection to daemon lost" and continue polling. Don't crash the UI.

    2. **Empty states**: When no jobs exist, show "No backup jobs configured. Use the CLI to add jobs." When a job has no history, show "No backup runs yet."

    3. **Loading states**: Initial load shows "Loading..." spinner/text. Each section loads independently.

    4. **Same-origin**: All API calls use relative paths (`/api/v1/status` not `http://127.0.0.1:8099/api/v1/status`) since the dashboard is served from the same API server.

    5. **Duration formatting**: Go serializes time.Duration as nanoseconds (int64). Convert to human-readable (e.g., 30000000000 → "30s").

    6. **Trigger backup**: Show a brief "triggered" confirmation (maybe a toast) after clicking Run Backup. Disable the button briefly to prevent double-clicks.

    7. **History caching**: Fetch history per-job when the job card is rendered. Cache in memory. On auto-refresh, only re-fetch history for the job whose card is expanded.

    8. **Collapsible history**: Click a "Show History" toggle on the job card. Use CSS transitions for smooth open/close.
