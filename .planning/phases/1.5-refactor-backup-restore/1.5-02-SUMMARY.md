# Phase 1.5 Plan 02: S3 Lifecycle API, Retention Enforcement & Batch Object Lock

## One-liner

Replaced stub `SetLifecyclePolicy`/`GetLifecyclePolicy` with real `PutBucketLifecycleConfiguration`/`GetBucketLifecycleConfiguration` SDK calls; rewrote `applyRetention()` to actually delete expired objects via `DeleteObject(BypassGovernanceRetention: true)` for GOVERNANCE mode; changed batch uploads from `PutObject` to `PutObjectWithLock` to respect job Object Lock configuration.

## Tasks Executed

### Task 1: Implement real S3 lifecycle API

- **File:** internal/s3client/client.go
- **Actions:**
  - Replaced stub `SetLifecyclePolicy` (printed manual instructions) with real `PutBucketLifecycleConfiguration` call using `s3types.LifecycleRule` with `backups/` prefix filter
  - Replaced stub `GetLifecyclePolicy` (returned static message) with real `GetBucketLifecycleConfiguration` call; handles `NoSuchLifecycleConfiguration` error code via `smithy.APIError.ErrorCode()` check, returning empty string for absent policies
  - Added `"encoding/json"`, `"errors"`, and `"github.com/aws/smithy-go"` imports
- **Verification:** `go build ./...` compiles, `grep` confirms both `PutBucketLifecycleConfiguration` and `GetBucketLifecycleConfiguration` present
- **Commit:** `4139953`

### Task 2: Fix retention policy to actually delete expired objects

- **File:** internal/backup/engine.go
- **Actions:**
  - Changed `applyRetention` signature from `func(job)` to `func(ctx context.Context, job) error`
  - For expired runs before the retention cutoff date: iterates entries via `GetEntriesForRun()`, calls `DeleteObject(s3Key, true)` with `BypassGovernanceRetention: true` for GOVERNANCE mode objects
  - For COMPLIANCE mode: logs a warning and skips deletion
  - Tracks and logs count of deleted expired objects
  - Updated call site in `RunBackup` (Step 7) to use `context.Background()` and log errors as warnings
- **Verification:** `go build ./...` compiles, `grep` confirms `DeleteObject` in engine.go, no `"S3 lifecycle will handle"` text present
- **Commit:** `8377faf`

### Task 3: Fix batch upload to use Object Lock

- **Files:** internal/s3client/batch.go, internal/backup/engine.go
- **Actions:**
  - Changed `BatchBuilder.Upload` signature: `(ctx, client)` → `(ctx, client, lockMode string, retentionDays int)`
  - Replaced `client.PutObject()` with `client.PutObjectWithLock()` for both batch archive and manifest upload
  - Updated engine.go caller to pass `job.ObjectLockMode` and `job.RetentionDays`
- **Verification:** `go build ./...` compiles, `grep` confirms 2x `PutObjectWithLock` in batch.go
- **Commit:** `98b5ec0`

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] SDK type references used wrong package**
- **Found during:** Task 1
- **Issue:** The plan's sample code used `s3.LifecycleRule`, `s3.LifecycleRuleFilter`, etc., but these types live in `s3types` (the `s3/types` subpackage alias). The plan correctly noted `s3types` was already imported but the sample code used the wrong prefix.
- **Fix:** Changed all lifecycle types to use `s3types.` prefix (`s3types.LifecycleRule`, `s3types.LifecycleRuleFilter`, `s3types.LifecycleExpiration`, `s3types.NoncurrentVersionExpiration`, `s3types.BucketLifecycleConfiguration`).
- **Files modified:** internal/s3client/client.go
- **Commit:** `4139953` (included in initial commit)

**2. [Rule 1 - Bug] NoSuchLifecycleConfiguration is not a Go type in this SDK version**
- **Found during:** Task 1
- **Issue:** The type `s3types.NoSuchLifecycleConfiguration` does not exist as a Go type in `aws-sdk-go-v2/service/s3@v1.100.0`. It only appears in doc comments. Using `errors.As` with this type produced a compile error.
- **Fix:** Replaced with `smithy.APIError` interface check: `errors.As(err, &apiErr) && apiErr.ErrorCode() == "NoSuchLifecycleConfiguration"`. Added `"github.com/aws/smithy-go"` import.
- **Files modified:** internal/s3client/client.go
- **Commit:** `4139953`
