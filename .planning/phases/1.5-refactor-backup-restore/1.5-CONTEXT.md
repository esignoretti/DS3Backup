# Phase 1.5 — Refactor Backup & Restore

## Phase Goal

Fix all documented tech debt, known bugs, performance bottlenecks, and missing features in the backup and restore pipeline. This is an inserted phase between Phase 1 (Foundation & Restore) and Phase 2 (Scheduling & Server).

## Decisions

### D-01: Remove duplicate DR backup call
Remove the second `createDisasterRecoveryBackup()` call at `internal/backup/engine.go:252-259`. The first call at line 237 is sufficient.

### D-02: Fix Object Lock mode hardcoded to GOVERNANCE
In `internal/s3client/client.go:124`, replace the hardcoded `"GOVERNANCE"` string with the `mode` parameter that is already passed to `PutObjectWithLock()`.

### D-03: Fix retention policy (currently no-op)
`applyRetention()` in `internal/backup/engine.go:394-411` marks runs for deletion but never deletes anything. Implement actual S3 object deletion for expired backups. For GOVERNANCE mode, use `DeleteObject` with `BypassGovernanceRetention: true`. For COMPLIANCE mode, log a warning indicating objects cannot be deleted within retention period.

### D-04: Remove duplicate badger v3 dependency
Remove `github.com/dgraph-io/badger/v3` from `go.mod` and run `go mod tidy`. Only v4 is used in source code.

### D-05: Consolidate duplicate restore worker patterns
Refactor `internal/restore/engine.go` to extract a shared `runRestorePipeline(entries, opts, tracker)` private method that all public restore entry points delegate to. Eliminate ~400 lines of near-duplicate code across `Restore()`, `RestoreWithProgress()`, `RestoreEntries()`, and `ResumeRestore()`.

### D-06: Consolidate duplicate formatting utils
Move `formatBytes()` and `formatDuration()` to a shared `internal/util/format.go` package. Update all callers (`internal/cli/utils.go`, `internal/restore/progress.go`, `internal/backup/engine.go`) to import from the shared location.

### D-07: Remove obsolete RebuildEngine stubs
Remove `RebuildEngine` struct (`internal/recovery/rebuild.go:175-184`), `RebuildIndex()`, `IsRetired()`, `DiscoverJobs()`, and `downloadJobMetadata()`. These are kept for "API compatibility" but all return stubs/empty results.

### D-08: Fix config save on failed backup
In `internal/cli/backup.go:134-137`, only update `job.LastRun` and call `cfg.SaveConfig()` when `err == nil`. Currently, `LastRun` is updated even on failed backups.

### D-09: VERSION file consolidation
Set source of truth to `internal/cli/version.go:4` (currently `0.0.7`). Update `VERSION` file and `README.md` header to match.

### D-10: Implement S3 lifecycle API
Replace stub implementations of `SetLifecyclePolicy()` and `GetLifecyclePolicy()` in `internal/s3client/client.go` with real AWS SDK v2 API calls (`PutBucketLifecycleConfiguration`, `GetBucketLifecycleConfiguration`).

### D-11: Implement index rebuild
Implement the TODO in `internal/cli/index.go:144` — scan S3 objects for `backups/<jobID>/` prefix, parse batch manifests to reconstruct file entries, and save to the local BadgerDB index.

### D-12: Fix batch upload skipping Object Lock
In `internal/s3client/batch.go:106`, change `client.PutObject()` to `client.PutObjectWithLock()` using the job's Object Lock mode and retention days. Require the `JobID` to look up the object lock config, or accept lock mode/retention as parameters.

### D-13: macOS BadgerDB lock issue (accept)
Document in `KNOWN_ISSUES.md` that this is unfixable without major architecture changes (upstream BadgerDB issue). No code change.

## The Agent's Discretion

- Retention policy: Whether to delete expired backup objects directly or rely on S3 lifecycle rules. Recommend: use direct deletion for GOVERNANCE mode, lifecycle for broader cleanup. Implement both: direct deletion in `applyRetention()`, and lifecycle API in `SetLifecyclePolicy()`.
- Formatting utils package naming: `internal/util/format.go` is preferred over `pkg/util` to keep it internal.
- Index rebuild strategy: Scan S3 for batch manifests + individual file objects, reconstruct `FileEntry` records from manifest metadata strutin.
- Version: Use `0.0.7` from `version.go` as source of truth (reflects post-Phase-2 state).
- Restore pipeline extraction: Extract a shared `runRestorePipeline` that accepts entries, options, and an optional progress tracker. `Restore()` gets full delegation, `RestoreWithProgress()` and `RestoreEntries()` delegate with tracker. `ResumeRestore()` delegates to the same pipeline but creates entries from `FileState`.

## Deferred Ideas

None.
